## 一、 技术栈选型与职责分工

系统采用了“前后端分离 + 业务算法分离”的核心思想，让 Java 负责高并发与工程规范，Python 负责 AI 生态与算法演进。

| **模块**         | **技术栈**                             | **核心职责**                                                 |
| ---------------- | -------------------------------------- | ------------------------------------------------------------ |
| **流量与业务层** | Java (Spring Boot)                     | 统一入口、JWT 用户鉴权、黑名单与流控、状态维护、对外接口暴露。 |
| **推理与算法层** | Python (FastAPI + LangChain)           | 负责 RAG 推理链路、多路召回、大模型（LLM）流式对接、Agent 决策。 |
| **消息中间件**   | RabbitMQ / Kafka                       | 削峰填谷，异步解耦核心问答流与长文本持久化。                 |
| **存储体系**     | Redis + PostgreSQL (pgvector) + 向量库 | **Redis**：缓存 Session 及滑动窗口内的短期对话历史。 **PostgreSQL**：持久化全量用户信息，并利用 `pgvector` 实现长期记忆向量检索。 **向量数据库**：存储本地 RAG 专属领域知识切片。 |

## 二、 三路召回（Three-Way Retrieval）架构设计

当用户发起一个提问时，Python 端的 Agent 系统为了保障回答的精准度与记忆连续性，会同时启动三路召回：

1. **第一路：短期记忆召回（当前上下文）**
   - Java 端直接从 **Redis** 中读取滑动窗口内（如最近 5 轮）的对话历史，打包随请求发送给 Python，作为大模型的当前对话背景。
2. **第二路：知识库召回（RAG 核心）**
   - Python 端对问题进行 Embedding，去 **向量数据库** 检索与之相关的本地垂直领域知识切片。
3. **第三路：长期记忆召回（跨越过期的记忆）**
   - **核心极客设计**：当滑动窗口过期的记忆（陈旧对话）需要被唤醒时，Python 通过 **MCP（Model Context Protocol）协议** 调用 Java 端的服务。Java 内部利用 **PostgreSQL (pgvector)** 进行历史对话的向量相似度检索，并将匹配到的历史记忆“反思与压缩”后返回给 Python Agent。

## 三、 完整的数据流向与持久化链路

为了极致优化用户的 **TTFT（首字响应时间）**，整个问答与写回流程采用了全链路异步化流式设计：

1. **鉴权入口**：前端携带 JWT 请求 Java，Java 验证通过并从 Redis 捞出短期上下文。
2. **流式转发（跨语言）**：Java 将请求打给 Python 端的 FastAPI，Python 启动三路召回并调用大模型，以 `StreamingResponse` 形式将字节流吐回给 Java。
3. **透明传输与内存累加**：Java 收到字节流后立即通过 **SSE (Server-Sent Events)** 转发给前端（用户几乎零延迟看到字蹦出来）。与此同时，Java 在内存中使用 `StringBuilder` 暗中拼接完整的回答文本。
4. **异步双写持久化（流结束触发）**：
   - **写 Redis**：Java 立即将“当前问题 + 完整回答”追加到 Redis 队列，更新该用户的最新滑动窗口上下文。
   - **写 PGSQL**：Java 将全量对话数据封装为消息丢进 **RabbitMQ**，业务线程立刻安全释放。随后，MQ 消费者异步监听消息，通过 **MyBatis** 平滑地将大文本和问题向量写入 **PostgreSQL (pgvector)**。



额外补充

用 **Java 的高性能网络能力（如 Netty, Spring WebFlux）搭一个中央 AI 网关**。 这个网关在服务层做的事情，正是你之前思考的：

- **动态流式路由**：统一拦截所有业务线的请求，自动把静态的 Prompt 格式化，甚至在网关层做安全审计（防止用户诱导 AI 吐出敏感词）。
- **感知推理层的状态**：Java 网关需要实时知道 Python 推理集群（vLLM）的负载情况，利用类似 **流式反压（Reactive Streams Backpressure）** 的机制，在 GPU 快要爆显存时，在 Java 端就进行排队和限流。
