### 一、 项目工程结构规范

后端项目应遵循**分层架构（Layered Architecture）**，这种方式能够显著解耦业务逻辑与底层基础设施。

- **Controller 层（接口层）：** 负责参数校验（DTO）、HTTP 请求/响应封装，严禁在 Controller 中编写复杂的业务逻辑。
- **Service 层（业务逻辑层）：** 处理核心逻辑，是系统的“大脑”。推荐使用接口隔离（Interface/Implementation），方便后续 Mock 或扩展。
- **Repository/DAO 层（数据访问层）：** 仅负责与数据库（MySQL/Redis/VectorDB）交互，使用 ORM 或 SQL 模板。
- **Config 层：** 存放全局配置类（如 DataSource 配置、Redis 客户端初始化）。

> **最佳实践建议：**
>
> 无论是使用 Java (Spring Boot) 还是 Python (FastAPI)，都应保持各层的**职责单一**。例如在 Python 中，可以使用 `schemas.py` 定义 Pydantic 模型，在 Service 层处理 AI Agent 的推理逻辑，而不是把 prompt 模板直接写在接口里。

### 二、 配置管理规范（12-Factor App 原则）

**绝对禁止将数据库密码、API Key 等敏感信息硬编码在代码中。**

- **环境隔离：** 使用不同的配置文件（如 `application-dev.yaml`, `application-prod.yaml`）区分环境。
- **环境变量（Environment Variables）：** 敏感配置（DB 用户名、密码、Redis 地址）应通过环境变量注入，或者使用配置中心（如 Nacos, Apollo, 或简单的 `.env` 文件）。
- **配置层级：** 默认配置 < 环境变量 < 命令行参数。

### 三、 数据库连接与命名规范

#### 1. MySQL 连接规范

- **连接池（Connection Pooling）是强制的：** 严禁手动创建连接。
  - **Java (Spring Boot):** 默认使用 **HikariCP**，它是目前性能最好的连接池。
  - **Python (FastAPI):** 使用 `SQLAlchemy` 或 `Tortoise-ORM` 的异步连接池，确保在 `lifespan` 事件中初始化连接池，在关闭时显式释放。
- **超时策略（Timeouts）：** 设置合理的 `connect_timeout` 和 `socket_timeout`。长时间阻塞的查询会导致系统雪崩。
- **读写分离：** 针对高并发场景，提前预留读写分离架构的设计（即区分 Master 与 Slave 节点）。

#### 2. Redis 键名规范 (Namespace)

Redis 是扁平化的 KV 结构，容易发生 Key 冲突，必须使用冒号 `:` 分隔的命名空间。

- **推荐格式：** `{业务模块}:{功能场景}:{唯一标识}`
  - `user:session:10023`
  - `agent:task:status:task_id_888`
  - `rank:daily:20260421`
- **关键点：**
  - 不要使用过长的 Key 名，避免占用内存。
  - **Hashtag 使用：** 如果需要使用 Redis Cluster，通过 `{...}` 将同一个业务相关的 Key 放在同一个 Slot 中，以支持 multi-key 操作。

### 四、 API 设计规范 (RESTful 标准)

参考 RESTful 规范，你的 API 应当对客户端是“可预测的”。

| **HTTP 方法** | **含义** | **示例**             |
| ------------- | -------- | -------------------- |
| **GET**       | 获取资源 | `GET /agents/101`    |
| **POST**      | 创建资源 | `POST /agents`       |
| **PUT**       | 整体更新 | `PUT /agents/101`    |
| **PATCH**     | 局部更新 | `PATCH /agents/101`  |
| **DELETE**    | 删除资源 | `DELETE /agents/101` |

#### 1. 统一响应格式

无论操作成功与否，返回的 JSON 结构应当统一，方便前端或 SDK 调用。

JSON

```
{
  "code": 200,          // 业务状态码
  "msg": "success",     // 错误信息
  "data": { ... },      // 返回的数据载体
  "traceId": "abc-123"  // 用于链路追踪，排查问题非常重要
}
```

#### 2. 状态码使用

- **200 OK:** 请求成功。
- **201 Created:** 创建资源成功。
- **400 Bad Request:** 参数校验失败（如字段缺失）。
- **401 Unauthorized:** 未登录。
- **403 Forbidden:** 无权限。
- **404 Not Found:** 资源不存在。
- **500 Internal Server Error:** 服务器内部错误（通常伴随 `traceId`，不应暴露具体报错栈给前端）。
